1. LEGAL ACCEPTANCE ARCHITECTURE
==================================================

We need two separate acceptance points:

A) Signup-level acceptance
B) Checkout / purchase-level acceptance

These should not be treated as the same event.

--------------------------------------------------
A. SIGNUP-LEVEL ACCEPTANCE
--------------------------------------------------

Student signup must require a mandatory checkbox:
"I agree to the Terms & Conditions and Refund Policy"

Requirements:
- checkbox unchecked by default
- signup must fail if not accepted
- backend must validate this, not only frontend
- store acceptance metadata in a minimal clean way

Preferred lean implementation:
- add user-level legal acceptance fields such as:
  - legalAcceptedAt
  - legalVersion

If current architecture strongly benefits from separate fields, that is acceptable, but do not overcomplicate.

--------------------------------------------------
B. CHECKOUT / PURCHASE-LEVEL ACCEPTANCE
--------------------------------------------------

Before payment, checkout must require a mandatory checkbox:
"I have read and agree to the Terms & Conditions and Refund Policy applicable to this purchase"

Requirements:
- payment/order creation must fail if not accepted
- backend must validate this, not only frontend
- store purchase-level legal acceptance metadata

Preferred lean implementation:
- add order/purchase/payment-level fields such as:
  - legalAcceptedAtCheckout
  - legalVersionAtCheckout

This is important for future refund/audit disputes.
