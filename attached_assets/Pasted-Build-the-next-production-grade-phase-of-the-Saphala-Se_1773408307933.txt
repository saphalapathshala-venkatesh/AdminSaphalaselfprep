Build the next production-grade phase of the Saphala Self Prep / Admin system for Flashcards and Protected Content Access Gate, using the existing stack and current live schema as safely as possible. Follow the already locked architecture direction: do not make unnecessary schema changes or disruptive refactors. Keep DB changes minimal and only add what is truly required.

Core goal

Implement a premium, reusable flashcard system for the student-facing Self Prep app and admin authoring flow, plus a protected-content consent gate for HTML materials, flashcard decks, and protected PDFs.

1) Flashcard system — overall rules

Implement the flashcard experience with these locked UX rules:

Only one card at a time

Clean, minimal, high-clarity layout

First card in every deck is a Title Card by default

All cards must include a global branding header

Subject/topic/subtopic must not consume large card space

Subject/topic/subtopic should appear in a small header/breadcrumb area

Content area should maximize usable study space

Card backgrounds should stay mostly white/clean; subject color should be used as accent only

Same experience should work cleanly on website and mobile-responsive web

2) Global branding header on every card

Every flashcard must include this default branding header at the top:

Saphala logo on the left

To the right:

Saphala Pathshala

Your Success is Our Focus

This should be built into the base flashcard layout, not manually added by admin per card.

Design guidance:

slim header, not bulky

consistent on every card

neutral background

subject color accents should appear below this branding row, not replace it

8) Quiz card behavior

Implement Quiz Card with this logic:

question shown first

options shown

student selects answer

must click Submit Answer

only after submit should evaluation happen

only after submit should Flip Card become active

after flip, show explanation

no difficulty/easy-medium-hard controls here; this is for reinforcement/revision, not evaluation

Support at least:

single-correct MCQ initially
Architecture can remain extensible for future quiz subtypes.

9) Typed Fill-in-the-Blank card

Implement this as a separate card type, not as drag/drop or word bank.

Behavior:

learner types answer manually into input field

clicks Submit Answer

system validates at coding level

if correct, Flip Card becomes available

if wrong, allow retry

after flip, show explanation

Critical requirement:
For each blank, admin must be able to define:

one canonical answer

multiple accepted answer variants (e.g. 4–5 valid forms)

Validation rules:

normalize input

trim whitespace

lowercase by default

collapse extra spaces

optionally ignore punctuation where appropriate

compare only against admin-approved accepted variants

Support:

single blank

multi-blank architecture if feasible cleanly, but single blank minimum required now

Do not use loose AI grading for this. Use deterministic accepted-answer matching.

10) Matching card

Implement a matching card type.

Behavior:

student performs matching interaction

on submit, evaluate matches

show green for correct, red for wrong

after submit, show two actions:

Retry

Flip Card

do not force student to stay until fully correct

after flip, show correct answer/explanation

continue button after flip

Retry behavior:
Prefer keeping correct matches and resetting only wrong ones, if feasible cleanly.

11) Reorder card

Implement reorder/arrange-in-correct-order as a separate card type.

Behavior:

admin defines a sequence of items/steps

learner arranges items into order

on submit:

evaluate

show green/red feedback by position

show Retry and Flip Card

learner should not be forced to get all correct before explanation

flip should reveal correct order + explanation

12) Categorization drag-and-drop card

Implement this as a separate card type from reorder.

This card type should support:

left side or source pool = items

right side = category boxes

learner drags items into category boxes

Critical admin configurability:

admin chooses number of items

admin chooses number of categories

categories are dynamic, not fixed

item counts per category can vary

each item maps to a correct category

support text items first; architecture should allow image support later

Behavior:

submit required

after submit:

show green for correct

show red for wrong

show Retry and Flip Card

do not force learner to stay until everything is correct

flip should show correct categorization + explanation

architecture should be robust and flexible, not hardcoded to 3 items / 3 categories

Responsive behavior:

desktop can be two-column

mobile should stack cleanly

16) Protected Content Access Gate

Implement a reusable Protected Content Access Gate that appears before opening any protected:

HTML material

Flashcard deck

protected PDF

This must not be just a frontend popup. It should be meaningful and backend-recorded.

Required UX

Before access, show a modal/dialog with copyright and monitoring warning.

Use copy similar to:

Protected Content Notice
This content belongs to Saphala Pathshala and is provided only for your personal learning use.
Copying, sharing, recording, downloading, redistributing, or publishing this content without permission is prohibited.
All your activities inside Saphala may be monitored for security and content protection purposes. If misuse or redistribution is detected, your account may be suspended or blocked as per platform policy.

Checkbox:

I have read and agree to the Terms & Conditions

Buttons:

Accept and Continue

Cancel

User must explicitly accept before content loads.

Important:

do not preload full HTML material before acceptance

do not fetch full flashcard deck before acceptance

do not open/generate protected PDF URL before acceptance

Acceptance should happen first, then content access should proceed.

Use a simple terms version such as:

content_protection_terms_v1

Add only minimal schema support required.
